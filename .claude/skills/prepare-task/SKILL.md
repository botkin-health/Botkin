# prepare-task — взять задачу в работу и подготовить план

## Контекст репо (Botkin)

- **Трекер:** GitHub Issues через `gh`, репо `botkin-health/Botkin`. Конвенции — `docs/agents/issue-tracker.md`.
- **Проект (доска):** GitHub Project **Botkin #1** (owner `botkin-health`, id `PVT_kwDOEX3Lns4Bam1p`). Status-поле `PVTSSF_lADOEX3Lns4Bam1pzhVdNrw`, опции: **Todo** `f75ad846` · **In Progress** `47fc9ee4` · **In Review** `c02f7d5b` · **Done** `98236657`. Все задачи заводим в этот проект. Статус **In Review** выставляет complete-task при создании PR — prepare-task его не трогает.
- **Базовая ветка:** `dev` (фич-ветки растут от `origin/dev`; в `main` льётся отдельно).
- **Worktree:** `.claude/worktrees/<ветка, `/` → `+`>` (напр. ветка `fix/foo` → `.claude/worktrees/fix+foo`).
- **Ветки:** `feat/ fix/ chore/ refactor/ test/ docs/` + короткий слаг. Номер issue в имя ветки НЕ кладём — связь через PR (`Closes #N`).
- **Коммиты:** conventional, **на русском**, со scope (напр. `fix(nutrition): …`). Атомарные, со ссылкой на issue.
- **Проверки (gate):** `ruff check .` · `ruff format --check .` · `PYTHONPATH=. pytest tests/ --ignore=tests/integration --ignore=tests/test_nutrition_parsing.py`. (mypy сконфигурён, но в CI не гоняется — опционально.)
- **Язык общения и записей в трекер:** русский.

Любой шаг упал/неоднозначен → **СТОП**, спросить пользователя. Не угадывать.

## Шаги (строго по порядку)

### 0. Нет issue → завести
Если задача дана текстом без номера:
- Определить тип: **баг** → ветка `fix/…`, лейбл `bug`; **фича** → ветка `feat/…`, лейбл `enhancement`; **идея/вопрос** → лейбл `enhancement`/`question` (метки `idea` в репе нет).
- **Заголовок** — по конвенции (см. ниже «Конвенция заголовков»).
- Тело = **постановка/проблема**, НЕ план.
- Показать черновик (title + body + label) → дождаться подтверждения → создать:
  ```bash
  gh issue create --title "<≤5 слов>" --label <label> --body "$(cat <<'EOF'
  <постановка/проблема>
  EOF
  )"
  ```
- Сразу добавить в проект **Botkin #1** со статусом **Todo** (см. «Доска» ниже).

### 1. Найти задачу и собрать контекст
`gh issue view <N> --comments` (+ labels/assignees/author/createdAt). Дать краткое резюме: title, state, labels, суть, ключевые комментарии.

### 1.5 Гейт метаданных (обязателен до взятия в работу)
Пройти и выдать **сводку одним списком**:
- **Лейблы** — если пусто, проставить по типу (`bug`/`enhancement`/`documentation`).
- **Проект** — issue в проекте Botkin #1? Если нет — добавить (идемпотентно, см. «Доска»).
- **Milestone** — если в репе ведётся; иначе пропустить.
- **Протухание** — `createdAt` > 30 дней → предупредить, спросить актуальность.
- **State** — если `CLOSED` → **СТОП** (задача закрыта).
- **Качество описания** — пустое/тривиальное → **СТОП**: попросить нормальную постановку + критерии приёмки, записать их в issue (`gh issue comment`/`gh issue edit`).

### 2. Взять в работу
- `gh issue edit <N> --add-assignee @me` (чужой ассайн не перехватывать без подтверждения).
- Выставить статус **In Progress** (`47fc9ee4`) на доске Botkin #1 (см. «Доска»).

### 3. Создать worktree от свежего origin/dev
```bash
git fetch origin dev
git worktree add .claude/worktrees/<slug> -b <feat|fix|…>/<slug> origin/dev
```
Сообщить путь к worktree. **Самому туда не `cd`**, в основном чекауте `checkout` не делать.

### 4. План — через `/everything-claude-code:plan`
Требование к структуре: план разбит на **фазы**, и КАЖДАЯ фаза заканчивается:
- **Fix** — прогон gate-проверок (`ruff check .` + `ruff format --check .` + pytest) до зелёного;
- **Commit** — атомарный conventional-коммит (русский) со ссылкой на issue (`… (#N)`);
- **Report** — итог фазы комментарием в issue (`gh issue comment <N>`).

### 4.4 Опц. ресёрч — `/deep-research`
Если без внешнего знания план мутный. Результат → `docs/researches/YYYY-MM-DD-<тема>.md` (папку создать при первом использовании).

### 4.5 Прожарка плана — `/grill-with-docs`
Против доменной модели: `CLAUDE.md`, `docs/ai_context/01_architecture.md`, ADR `docs/architecture/decisions/`, `docs/agents/domain.md`.

### 5. Показать план и ждать «да»
Комментарий в issue НЕ постить без явного подтверждения.

### 6. Запостить план
После «да» — `gh issue comment <N> --body "<план>"`.

## Справка

### Конвенция заголовков issue
- **≤ 5 слов**, но из заголовка сразу ясно **область + суть** задачи.
- Формат «по сути»: `<область>: <что делаем>` или короткая именная группа. Без «фиксим/добавить/нужно» — это шум.
- Детали, шаги, воспроизведение, критерии приёмки → в **тело**, не в заголовок.
- Заголовок issue — человекочитаемая фраза (НЕ conventional-commit; conventional — это для веток/коммитов/PR).
- Если суть в 5 слов не лезет — задача слишком крупная: предложить разбить на несколько issue.

| Задача | ✅ Заголовок | ❌ Плохо |
|---|---|---|
| фича онбординга в боте | `онбординг пользователей в боте` | `Сделать онбординг новых юзеров через мастер-визард` |
| набор UI-правок админки | `UI улучшения админки` | `Фиксы` / `мелочи по админке` |
| баг логирования еды | `еда: уточнение пишется новым приёмом` | `Не работает добавление еды` |
| импорт Apple Health | `Apple Health: self-serve подключение` | `эпл хелф` |

### Доска: добавить задачу и выставить статус
```bash
# добавить issue в проект Botkin #1 (идемпотентно; вернёт item id)
ITEM_ID=$(gh project item-add 1 --owner botkin-health --url <issue-url> --format json --jq '.id')
# статус: Todo (f75ad846) / In Progress (47fc9ee4) / In Review (c02f7d5b) / Done (98236657)
gh project item-edit --id "$ITEM_ID" --project-id PVT_kwDOEX3Lns4Bam1p \
  --field-id PVTSSF_lADOEX3Lns4Bam1pzhVdNrw --single-select-option-id <option-id>
```

## Чего НЕ делать
- Не заводить задачу мимо проекта **Botkin #1**; статусы только Todo/In Progress/In Review/Done (другие не выдумывать). In Review ставит complete-task, не prepare-task.
- Не пушить ветку/каждый коммит в origin сразу (пуш — в complete-task).
- Не делать `cd`/`git checkout` в основном чекауте — только `git worktree add`.
- Ветка только от `origin/dev` (не от локального dev, не от main).
- Не планировать до взятия в работу и создания worktree.
- Не пропускать гейт метаданных, добавление в проект и прожарку.
- Не постить план без «да».
- Не трогать чужие ассайни/приоритет/лейблы сверх необходимого.
