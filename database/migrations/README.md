# Database migrations

> ⚠️ **Исторический архив (#83).** Схемой управляет **Alembic** (`database/alembic/`).
> Эти `.sql` — лог эволюции схемы до перехода на Alembic; они **не накатываются**
> автоматически. Baseline (вся текущая схема прода) — в ревизии
> `database/alembic/versions/711fd5e3f1e8_baseline_schema.py`. Новые изменения схемы
> делаются ревизиями Alembic, не файлами здесь. Подъём БД с нуля: `alembic upgrade head`.
> Старый bootstrap-файл `init.sql` удалён — его роль выполняет baseline-ревизия.

Каждый `.sql` файл здесь — один шаг эволюции схемы Postgres (исторически). По сути это
«пошаговая инструкция», как из пустой БД получить ту, что крутилась на проде до Alembic.

## Что такое миграция

Миграция — это SQL-скрипт, описывающий изменение схемы:
добавить колонку, создать таблицу, накатить индекс, починить тип. Пример:

```sql
ALTER TABLE users
    ADD COLUMN IF NOT EXISTS agent_review_consent BOOLEAN NOT NULL DEFAULT TRUE;
```

Хранить такие скрипты в репозитории нужно, чтобы:
1. Видеть **историю** — что и когда меняли в схеме.
2. **Поднять с нуля** новую копию БД (dev / staging) одной командой.
3. Не зависеть от того, что что-то «накликали руками» в проде.

## Соглашения

- Имя файла: `add_<что>.sql` / `fix_<что>.sql` / `drop_<что>.sql`.
- Идемпотентность: всегда `IF NOT EXISTS` / `IF EXISTS`, чтобы повторное
  применение не падало. У нас пока нет миграционного фреймворка, который
  отслеживает «накатано или нет», — спасает идемпотентность.
- В начале файла — короткий комментарий: зачем, когда применено,
  ссылка на ADR/issue если есть.
- НЕ редактировать уже применённые миграции. Если что-то пошло не так —
  делать новую `fix_*` сверху.

## Как накатить на прод

```bash
# с маки, путь к серверу — см. scripts/fetch_remote_nutrition.sh
SSHPASS=$SERVER_PASSWORD sshpass -e ssh root@<server> \
  "docker exec -i healthvault_postgres psql -U healthvault -d healthvault" \
  < database/migrations/<filename>.sql
```

Либо вручную через `psql` на сервере. После применения — рестарт `healthvault_bot`,
если SQLAlchemy-модели тоже поменялись.

## Чего здесь нет (пока)

- **Down-миграций** (откат). Если нужно откатиться — пишем `fix_*.sql`
  с обратным `ALTER`. Подходит для текущей стадии «одна БД, один разраб».
- **Migration runner** (Alembic / Flyway). Когда заведём dev/staging — стоит
  внедрить, чтобы помнил, что уже применено.
- **Версионирования через timestamp в имени.** Тоже к будущему фреймворку.
