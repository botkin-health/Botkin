# ADR-0003: Alembic как фреймворк миграций БД

**Дата:** 2026-06-14
**Статус:** Accepted
**Автор:** Игорь Николаев
**Связи:** [#83](https://github.com/botkin-health/Botkin/issues/83), ресёрч `docs/researches/2026-06-14-alembic-vs-flyway.md`

## Контекст

Схема Postgres велась **вручную голым SQL** в `database/migrations/*.sql` без миграционного фреймворка: не было трекинга «накатано/нет» (спасала идемпотентность), down-миграций, версионирования; накат — `psql < file.sql` руками. Результат — рассинхрон: прод и dev разошлись (на dev висели ретайрнутые таблицы `user_products`), а `init.sql` устарел (8 таблиц из 15, без `share_token`/RLS) и тихо создавал кривую схему на свежих БД.

Проект движется к multi-user + dev/staging — ручной процесс стал источником ошибок.

## Решение

**Внедряем Alembic** (не Flyway/Liquibase). Обоснование — `docs/researches/2026-06-14-alembic-vs-flyway.md`: Alembic — чистый Python без JVM (Flyway тянет Java-рантайм в Docker), нативная связка с SQLAlchemy 2.0, `autogenerate` из моделей, поддержка async-движка. Для Python/SQLAlchemy-проекта — не близкий выбор.

Ключевые решения реализации:

1. **Источник истины схемы — Alembic + ORM.** Все таблицы описаны в `database/models.py`; изменения схемы — ревизиями Alembic. Старые `.sql` — исторический архив (не накатываются).
2. **ORM приведён 1:1 к реальной прод-схеме** (снят `pg_dump`, доведено до пустого `alembic check`): 7 недостающих таблиц заведены, `JSON→JSONB` (через `with_variant`, чтобы sqlite-тесты жили), unique-констрейнты и имена под прод.
3. **`NOT NULL`: зеркалим прод сейчас, ужесточаем позже.** Где прод nullable, а ORM был строже — ослабили ORM до nullable (чтобы baseline = точный стэмп прода без изменения схемы). Ужесточение `NOT NULL` — отдельной ревью-миграцией, после проверки/чистки NULL-строк.
4. **Baseline-ревизия = полная прод-схема + RLS.** `711fd5e3f1e8` поднимает БД с нуля (15 таблиц, индексы, роль `hv_app`, 8 политик, гранты). Существующие БД (prod, dev) **застэмплены** (`alembic stamp`) без пересоздания.
5. **`init.sql` удалён.** Bootstrap пустой БД теперь через `alembic upgrade head`; монтирования `docker-entrypoint-initdb.d` сняты.
6. **Деплой миграций — только GitHub Actions** (`migrate.yml`, `workflow_dispatch` env=dev/prod), с **обязательным `pg_dump`-бэкапом перед накатом** на любом стенде. Согласуется с переходом на GHCR pull-only деплой (`deploy-prod.yml`, deploy.sh удалён).
7. **CI round-trip** в `test.yml`: на эфемерном Postgres `upgrade head → check → downgrade base → upgrade head`.

## Последствия

**Плюсы:** воспроизводимая схема с нуля; трекинг применённых ревизий; autogenerate ловит дрейф модель↔БД; единый язык (Python); бэкап перед каждым накатом.

**Минусы / долги:**
- На dev осознанно оставлен дрейф — 2 ретайрнутые таблицы `user_products`/`user_product_variants` (прод их уже дропнул вручную). **Follow-up:** оформить их дроп отдельной ревью-миграцией (идемпотентный `DROP TABLE IF EXISTS` — на dev дропнет, на проде no-op). До тех пор `alembic check` на dev показывает эти 2 таблицы.
- **Follow-up:** ревью-миграция ужесточения `NOT NULL` (п.3).
- `migrate.yml` применим только после деплоя образа с alembic-кодом (бинарь alembic в образе есть, конфиг приедет с этой веткой).
- Роль `hv_app` создаётся в baseline без пароля (LOGIN) — пароль задаёт оператор вне миграции (секрет).
