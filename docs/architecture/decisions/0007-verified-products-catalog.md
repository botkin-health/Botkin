# 0007. Справочник проверенных продуктов: двухступенчатый матчинг, автонаполнение вместо CRUD

**Status:** Accepted
**Date:** 2026-07-03
**Deciders:** Александр Лысковский, Claude
**Context:** [#255](https://github.com/botkin-health/Botkin/issues/255)

## Проблема

Распознавание еды — чистый LLM-vision без памяти: один и тот же упакованный продукт каждый раз оценивается заново и с ошибками. Прецедент 03.07.2026: батончик Solvie Protein Barre (этикетка: 144 ккал / Б 16.7 / Ж 6.6 / У 4.5 / клетчатка 11.4 на 50 г) — бот записал Б 12 / У 10 / клетчатка 1.8 (белок −28%, клетчатка −85%). Для упакованных продуктов это систематическая, полностью устранимая ошибка.

## Решение

Таблица `verified_products` (Alembic-ревизия `verprod01`): этикеточные КБЖУ на 100 г + порция + barcode. `user_id NULL` = общая запись для всех, иначе личная; личная приоритетнее общей при матчинге.

**Двухступенчатый матчинг** — обе ступени, не одна:

1. **Промпт-инъекция** (`build_known_products_block`, топ-20 по `times_used`): LLM видит список проверенных продуктов пользователя и берёт цифры оттуда при распознавании. Мягкая ступень — LLM может проигнорировать.
2. **Детерминированный post-match** (`match_and_apply_verified_products` в `save_meal_to_db`, до `enrich_items_with_fiber`): точное совпадение нормализованного имени/алиаса/«бренд имя» → пересчёт КБЖУ из `*_per_100g` по весу (или `portion_g`). Жёсткая ступень — гарантия, даже если LLM ошибся.

**Наполнение — автоматическое**: LLM извлекает `product_label`, когда реально читает таблицу пищевой ценности (фото этикетки или точные цифры текстом); после сохранения приёма — одна кнопка «💾 Запомнить продукт». Плюс сид-скрипт `scripts/import/seed_verified_products.py` (общие записи).

## Ключевые trade-off'ы

- **Почему не только промпт:** LLM недетерминирован — цифры в промпте не гарантируют их применение. Post-match даёт детерминизм.
- **Почему не только post-match:** промпт-ступень помогает LLM правильно назвать и взвесить продукт (например, узнать порцию 50 г), что улучшает и сам матч.
- **Почему точное совпадение, а не substring:** substring по одному слову («батончик») давал бы ложные матчи обычной еды на справочные продукты. Fuzzy-матч сознательно отложен.
- **Почему автонаполнение, а не CRUD:** предыдущая инкарнация — `/my_products` (таблицы `user_products`/`user_product_variants`, ручные команды) — удалена 21.04.2026 с **0 строк за всё время жизни**. Ручное заполнение справочников в этом продукте мертво.
- **RLS-отклонение:** политика чтения `user_id IS NULL OR user_id = app.user_id` (WITH CHECK — только свои строки) — общий справочник по дизайну виден всем через агентский слой hv_app; строгий `user_isolation` других таблиц прятал бы общие записи.

## Отвергнутые альтернативы

- **Внешние barcode-базы (OpenFoodFacts)** — отложено; поле `barcode` заложено, интеграция отдельной задачей.
- **Кэш в `data/products.json`** (существующий `core/food/product_search.py`) — файл вне Postgres, не per-user, не подмешивается в промпт. Остаётся legacy-fallback до отдельной задачи по выпиливанию.
- **UI управления справочником** — не делаем, пока не доказано, что наполнение работает.

## Последствия

- Повторное фото известного продукта даёт этикеточные КБЖУ детерминированно.
- Промпт растёт на ≤2500 символов для пользователей со справочником (у Claude — вторым system-блоком без `cache_control`, кеш базового промпта не ломается).
