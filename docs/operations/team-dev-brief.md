## 2. Как сейчас код попадает на прод (текущий процесс)

```
Олег/Игорь вайбкодят у себя
        │  делают Pull Request (PR) на GitHub
        ▼
   GitHub автоматически гоняет тесты (CI)  ← это уже работает
        │
        ▼
   Александр смотрит PR и нажимает "Merge"  ← решение за ним
        │
        ▼
   Claude (или Александр) вручную деплоит:  ← это ПОКА вручную
   ./deploy.sh  →  заливает код на сервер + пересобирает Docker + рестарт
```

**Вручную деплой** = скрипт `deploy.sh` с мака: он копирует код на сервер (`rsync`),
пересобирает Docker-образ и перезапускает бота. Пароль сервера берётся из `.env`.
Без этого шага изменения на проде НЕ появляются (мерж в GitHub ≠ деплой).

## 3. Что УЖЕ автоматизировано ✅ и чего НЕ хватает ❌

| | Статус |
|---|---|
| ✅ **Тесты на каждый PR** | Есть. `.github/workflows/test.yml` гоняет ~540 pytest-тестов на каждый PR и push в main. Поэтому у PR-ов видно «галочку» прошёл/не прошёл |
| ❌ **Авто-деплой после мержа** | НЕТ. После мержа кто-то руками запускает `deploy.sh`. **Это и предлагает автоматизировать Игорь Н.** |
| ❌ **Защита ветки main** | НЕТ. Сейчас технически можно влить PR даже с красными тестами или запушить напрямую в main. Стоит включить «нельзя мержить без зелёных тестов + ревью» |
| ❌ **Dev/тестовый сервер** | НЕТ. Есть только прод. Ребятам негде свободно проверять свой код «вживую», кроме как на боевом |

## 4. Куда хотим прийти (целевая картина)

```
            ┌─────────────────────────────────────────────┐
            │  DEV-окружение (песочница)                   │
Олег/Игорь →│  отдельный бот + отдельная БД                │  ← катят свободно,
            │  ломают и проверяют что навайбкодили          │     БЕЗ риска проду
            └─────────────────────────────────────────────┘
                              │  когда готово — PR в main
                              ▼
            ┌─────────────────────────────────────────────┐
            │  GitHub: тесты (CI) + ревью + защита main     │  ← гейт качества
            └─────────────────────────────────────────────┘
                              │  мерж → авто-деплой
                              ▼
            ┌─────────────────────────────────────────────┐
            │  ПРОД  (живые данные семьи, реальный бот)     │  ← сюда руками никто
            │  деплой делает машина, не человек             │     не лезет
            └─────────────────────────────────────────────┘
```

**Главный принцип:** новички НЕ получают прямой доступ катить на прод руками.
Прод обновляется сам при мерже зелёного PR. А «поиграть и проверить» — в dev-песочнице.

## 5. Бэкапы (чтобы не бояться экспериментов)

- Каждую ночь в 03:30 — дамп БД (`pg_dump`) в `/opt/backups` **+ копия на Google Drive** (схема «дед-отец-сын»: дневные/недельные/месячные).
- Раз в месяц — авто-проверка, что бэкап реально восстанавливается.
- То есть данные защищены — это даёт право экспериментировать смелее.

## 7. Словарик (для своих)

- **PR (Pull Request)** — «предложение изменений». Не меняет прод, пока не смержат.
- **Merge (мерж)** — принять PR в главную ветку `main`.
- **CI** — робот на GitHub, который автоматически гоняет тесты на каждый PR.
- **Deploy (деплой)** — выложить код на боевой сервер, чтобы изменения заработали у пользователей.
- **CI/CD** — когда и тесты, и деплой делаются автоматически по триггеру (мержу).
- **Dev / staging** — тестовое окружение-двойник прода, где можно ломать без последствий.
- **Branch protection** — правила GitHub: например «нельзя мержить в main без зелёных тестов».
