# 🛠️ Botkin Health Export — как собрать/пересобрать команду iOS Shortcuts

> Технический блюпринт бесплатной команды «Botkin Health Export» (iOS «Команды»).
> Нужен, потому что `.shortcut` Apple отдаёт **зашифрованным** — исходник нельзя
> закоммитить в git, распространение только через подписанную iCloud-ссылку.
> Этот файл — единственный источник правды по внутренней структуре команды.
> Пользовательская инструкция «как поставить готовую» — в [apple-health.md](apple-health.md).
>
> Структура сверена с реальной командой «Botkin Health Export 1» (скриншоты, 19.07.2026).

## Что делает команда

При каждом открытии Telegram (экран разблокирован → Apple Health доступен) команда
читает метрики **за сегодня**, собирает плоский JSON и шлёт:

```
POST https://botkin.health/apple_health
Authorization: Bearer hvt_<uid>_<hex>
Content-Type: application/json
```

Приёмник — `telegram-bot/webhook/apple_health.py` (endpoint v1, схема
`AppleHealthPayload`). **Сервер уже принимает все поля ниже — доработок бэкенда не
нужно, только пересборка команды.** Дедуп по дате: повторная отправка за день
обновляет запись.

## Фактическая структура (что уже собрано)

Порядок действий в текущей команде:

1. **Комментарий** — инструкция вставить ключ.
2. **Текст** = `Bearer hvt_…` (весь ключ целиком, вместе со словом Bearer).
3. **Задать переменную** `token` = Текст. Эта переменная кладётся прямо в заголовок
   `Authorization` — то есть «Bearer » уже внутри токена, отдельно его не дописывать.
4. **Форматировать дату:** Текущая дата → строка `yyyy-MM-dd` → ключ `date`.
5. Дальше — блоки метрик (см. таблицы) → **Получить содержимое URL** (POST).

## Общие правила (важно)

1. **Разрешение на объём:** Настройки → Команды → Дополнения → включить
   «Разрешить публикацию большого количества данных» (команда читает сотни сэмплов
   пульса за день).
1a. **Доступ к типам Health (частая боль новых юзеров).** HealthKit требует согласия
   на **каждый тип** отдельно (privacy-механика Apple). Массовая выдача:
   - при первом запуске готовой команды iOS обычно показывает **один общий лист** со
     всеми типами + «Включить все» → «Разрешить» (пошаговые запросы вылезают, если
     команду собирали/редактировали инкрементально);
   - надёжный batch: **Здоровье → профиль → «Приложения» → «Быстрые команды» →
     «Включить все категории»** (приложение появляется там после первого запроса).
   > ⏳ Точные формулировки UI сверяются на телефоне (19.07.2026) — уточнить раздел
   > «Разрешения» в [apple-health.md](apple-health.md) по факту.
2. **Единица задаётся прямо в «Найти Данные Здоровья» → поле «Единица».** Так и
   сделано в команде: distance → км, active → ккал, HRV → мс, SpO₂ → %, пульс →
   число/мин. Ставим нужную единицу в каждом Find (колонка «Единица» в таблицах) —
   **ручная конверсия не нужна**, значение уже в целевых единицах. Fallback
   «Число × K» — только если для типа нет подходящей единицы в списке.
3. **Сумма vs Среднее — разное группирование:**
   - **Суммы** (шаги, дистанция, калории, этажи): «Единица группирования» = **День**,
     «Заполнить отсутствующие» = **ВКЛ** → «Вычислить **Сумма**».
   - **Средние** (пульс, HRV, SpO₂, дыхание, походка): «Единица группирования» =
     **Нет** → «Вычислить **Среднее**» → для целочисленных полей ещё «**Округлить**,
     режим Стандартный». Группирование «Нет» критично: иначе сэмплы суммируются и
     пульс приходит тысячами.
4. **Имена ключей JSON — строго английские snake_case из таблиц.** Опечатка → поле
   молча игнорируется (Pydantic отбрасывает лишнее — так сейчас теряется
   `resting_heart_avg`, см. «Баги» ниже).
   ⚠️ **Названия типов в пикере «Тип» — на языке системы (у русского iOS —
   русские).** Уже выбранные типы могут показываться по-английски (наследие
   авторской локали), но при выборе нового типа список русский. Соответствие —
   в разделе «Русские названия типов». Ключи JSON при этом всегда английские.
5. **Заголовки POST:** только `Authorization` (= перем. `token`) и `Content-Type:
   application/json`. Пустых заголовков не добавлять — дают «400 / cloudflare».

## Шаблон одного блока

```
1. Найти образцы здоровья (Найти Данные Здоровья)
     • Тип = <HealthKit-тип>
     • Начало = сегодня
     • Единица = <целевая единица из таблицы>
     • Единица группирования = День (для сумм) | Нет (для средних)
     • Заполнить отсутствующие = ВКЛ (для сумм)
     • (эпизодические: Сортировка = Дата окончания ↓, Ограничение = 1)
2. Вычислить (Статистика): Сумма | Среднее | Минимум | Максимум
3. (для целых) Округлить → Стандартный
4. Записать результат в поле JSON под нужным ключом
```

Эпизодические метрики (вес, состав тела, давление, VO₂max) оборачивать в «Если»
(количество образцов > 0) — иначе в дни без замера ключ не добавляется (не шлём
вчерашний вес под сегодняшней датой).

## 🟢 Тир 1 — базовые (8 уже есть + 4 добавить)

| Метрика | HealthKit-тип | Операция | Единица (в Find) | Ключ JSON | Статус |
|---|---|---|---|---|---|
| Шаги | Steps | Сумма | шт | `steps` | ✅ есть |
| Дистанция | Walking + Running Distance | Сумма | км | `distance_walking_km` | ✅ есть |
| Активные ккал | Active Calories | Сумма | ккал | `active_energy_kcal` | ✅ есть |
| Пульс средний | Heart Rate | Среднее + Округлить | число/мин | `heart_rate_avg` | ✅ есть |
| Пульс покоя | Resting Heart Rate | Среднее + Округлить | число/мин | `resting_heart_rate` | ⚠️ ключ исправить |
| HRV | Heart Rate Variability | Среднее + Округлить | мс | `hrv` | ✅ есть |
| SpO₂ | Oxygen Saturation | Среднее | % | `spo2_pct` | ✅ есть |
| Частота дыхания | Respiratory Rate | Среднее | число/мин | `respiratory_rate` | ✅ есть |
| **Пульс min** | Heart Rate *(тот же Find)* | **Минимум** + Округлить | число/мин | `heart_rate_min` | 🆕 добавить |
| **Пульс max** | Heart Rate *(тот же Find)* | **Максимум** + Округлить | число/мин | `heart_rate_max` | 🆕 добавить |
| **Этажи** | Flights Climbed | Сумма | шт | `flights_climbed` | 🆕 добавить |
| **Базальные ккал** | Resting Energy | Сумма | ккал | `basal_energy_kcal` | 🆕 добавить |

> ⚡ **Экономия:** `heart_rate_avg/min/max` — это **один** блок «Найти: Heart Rate,
> сегодня, группирование Нет» → три «Статистики» (Среднее/Минимум/Максимум) на его
> результате. Не искать пульс трижды.
>
> 💡 `basal_energy_kcal` (Resting Energy / «Энергия покоя») особенно важен: сервер
> берёт его как BMR для юзеров без Garmin
> ([apple_health.py](../../../telegram-bot/webhook/apple_health.py) →
> `bmr_calories`, приоритет Garmin > Apple). `active_energy_kcal` при этом идёт
> только в архив `raw_data` (в калорийный триплет не пишется — источник Garmin).
> Если типа «Resting Energy» нет в списке — попробуй «Basal Energy Burned».

## 🟡 Тир 2 — походка (gait)

Один блок Find на каждую: сегодня, Единица группирования = Нет, «Вычислить Среднее».

| Метрика | HealthKit-тип | Единица (в Find) | Ключ JSON |
|---|---|---|---|
| Скорость ходьбы | Walking Speed | км/ч | `walking_speed_km_h` |
| Длина шага | Walking Step Length | см | `walking_step_length_cm` |
| Двойная опора | Walking Double Support % | % | `walking_double_support_pct` |
| Асимметрия | Walking Asymmetry % | % | `walking_asymmetry_pct` |

> **VO₂max — НЕ собираем (решение 19.07.2026).** Apple считает его раз в несколько
> дней после уличной прогулки/пробежки → при фильтре «сегодня» поле почти всегда
> пустое. Плюс это единственная эпизодическая метрика без серверного guard'а
> (пишется в `raw_data` как есть) → пустой день слал бы `vo2_max: 0`. На today-модели
> бесполезен. Если когда-нибудь понадобится — фильтр «в последние 7 дней» +
> Сортировка «Дата окончания ↓» + Ограничение 1 (значение будет повторяться до
> пересчёта Apple).

## 🟠 Тир 3 — эпизодические (без «Если» — сервер сам отсекает пустые)

**«Если» НЕ нужен.** Тело POST — фиксированный JSON, из него нельзя условно убрать
поле (это умеет только конструкция через «Словарь», а она тут не нужна). В дни без
замера Find вернёт пусто → «Вычислить Среднее» вернёт `0`, а сервер его отбросит:

- `weight_kg`, `muscle_mass_kg`, `water` — пишутся только при `weight_kg > 30`;
- `body_fat_pct` — только вместе с весом (та же проверка);
- давление — только если пришли **обе** цифры.

Паттерн (как у средних): Find (сегодня, группирование = Нет) → «Вычислить Среднее»
→ поле в теле JSON.

| Метрика | HealthKit-тип | Единица (в Find) | Ключ JSON |
|---|---|---|---|
| Вес | Weight (Body Mass) | кг | `weight_kg` |
| % жира | Body Fat Percentage | % | `body_fat_pct` |
| Безжировая масса тела | Lean Body Mass | кг | `muscle_mass_kg` |
| Давление верхнее | Blood Pressure Systolic | мм рт. ст. | `blood_pressure_systolic` |
| Давление нижнее | Blood Pressure Diastolic | мм рт. ст. | `blood_pressure_diastolic` |

> **Грабли:** переменных «Среднее» станет много — при добавлении поля в тело JSON
> выбирай «Среднее» **именно из нужного блока Find** (Shortcuts подписывает их по
> действию-источнику), иначе легко подставить вес в жир.
>
> **Проверка пустого дня:** прогони команду в день без взвешивания — POST должен
> по-прежнему вернуть `status: ok`. Если вдруг прилетит 422 (Среднее вернуло пусто,
> а не 0) — тогда только для проблемного поля обернуть в «Если > 0» через «Словарь».
>
> **`water_pct`** (процент воды) — в стандартном HealthKit **нет** такого типа
> (Mi-весы пишут его только в своё приложение, не в Apple Health) → **пропускаем**.

## 🔴 Тир 4 — хрупкие на бесплатном пути (по желанию, рекомендация — оставить HAE)

Технически возможны, но на «сегодня + открытие Telegram» ненадёжны: сон пересекает
полночь, и при утреннем открытии прошлая ночь может не попасть в окно «сегодня».
Полноценно их тянет HAE. Если всё же включаем:

| Метрика | HealthKit-тип | Как | Ключ JSON |
|---|---|---|---|
| Сон всего | Sleep Analysis, «Asleep» | сумма длительностей за последнюю ночь → часы | `sleep_hours` |
| — фазы | Sleep Analysis (Deep/REM/Core/Awake) | по категориям, часы | `sleep_deep_h`, `sleep_rem_h`, `sleep_core_h`, `sleep_awake_h` |
| Температура запястья | Apple Sleeping Wrist Temperature | ночная, Среднее | `wrist_temperature` |

## Русские названия типов в пикере «Тип»

На русском iOS список типов в «Найти Данные Здоровья» русский. Ключи JSON остаются
английскими. Надёжный признак верного выбора — единица в поле «Единица».

| Ключ JSON | Тип в пикере (рус.) | Единица |
|---|---|---|
| `steps` | Шаги | шт |
| `distance_walking_km` | Дистанция ходьбы и бега | км |
| `active_energy_kcal` | Активная энергия / Активные калории | ккал |
| `basal_energy_kcal` | Энергия покоя | ккал |
| `flights_climbed` | Пройдено пролётов | шт |
| `heart_rate_avg/min/max` | Пульс | число/мин |
| `resting_heart_rate` | Пульс в покое | число/мин |
| `hrv` | Вариабельность сердечного ритма | мс |
| `spo2_pct` | **Кислород в крови** (= SpO₂, не VO₂max!) | % |
| `respiratory_rate` | Частота дыхания | число/мин |
| `walking_speed_km_h` | Скорость при ходьбе | км/ч |
| `walking_step_length_cm` | Длина шага при ходьбе | см |
| `walking_double_support_pct` | Двойная опора при ходьбе | % |
| `walking_asymmetry_pct` | Асимметрия при ходьбе | % |
| `vo2_max` | **Кардиовыносливость** | мл/(кг·мин) |
| `weight_kg` | Вес | кг |
| `body_fat_pct` | Процент жира в организме | % |
| `muscle_mass_kg` | Безжировая масса тела (Lean Body Mass) | кг |
| `blood_pressure_systolic` | Систолическое давление | мм рт. ст. |
| `blood_pressure_diastolic` | Диастолическое давление | мм рт. ст. |

> ⚠️ **VO₂max = «Кардиовыносливость»**, а не «VO2» (в русском пикере строки «VO2»
> нет). «Кислород в крови» — это SpO₂ (`spo2_pct`), другой показатель. Названия
> могут чуть отличаться по версии iOS — сверяй по единице.

## ⚠️ Баги/расхождения, найденные при сверке (19.07.2026)

1. **`resting_heart_avg` → `resting_heart_rate`.** В теле запроса ключ пульса покоя
   назван `resting_heart_avg`; endpoint ждёт `resting_heart_rate` и молча отбрасывает
   неизвестное поле → пульс покоя сейчас **не сохраняется** (в `activity_log` уходит
   только `heart_rate_avg` через fallback `resting_heart_rate or heart_rate_avg`).
   **Фикс:** переименовать ключ поля.

## Финальные действия (Получить содержимое URL)

- URL: `https://botkin.health/apple_health` · Метод: **POST**
- Заголовки: `Authorization` = перем. `token`, `Content-Type` = `application/json`
- Тело запроса: **JSON** — по одному полю на метрику (ключ = англ. имя, значение =
  результат «Статистики»/«Округления»). `date` = «Форматированная дата».
- (опц.) «Показать уведомление» с ответом сервера (`saved: […]`) — для отладки.

## Проверка после сборки

1. Запусти команду вручную (кнопка ▶) или открой Telegram.
2. Ответ сервера должен содержать `"status": "ok"` и список `saved`.
3. Сверь значения в боте (Mini-app → Дневник / Dashboard). Единицы уже заданы в
   Find, но на всякий случай глянь SpO₂, проценты походки, вес — если значение
   в ×100/÷100 раз мимо, поменяй «Единицу» в Find или добавь «Число × K».
