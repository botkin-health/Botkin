# Family-forward: фото еды + КБЖУ доверенным получателям

## Задача
Когда пользователь сохраняет приём пищи в боте, автоматически переслать **копию фото блюда + КБЖУ** заранее настроенным получателям (например, супруге и лечащему врачу-эндокринологу).

Кейс: Андрей (E16.1, демпинг-синдром) логирует еду → жена Алла и эндокринолог Романова видят, что и когда он ел, без ручной пересылки.

## Граница приватности (жёсткая)
Пересылается **ТОЛЬКО еда**: фото блюда + калории/белки/жиры/углеводы + название.
**НЕ пересылается** ничего медицинского: глюкоза/CGM, давление, вес, диагнозы, анализы, лекарства, дневник гипо. Получатель видит только рацион.

## Модель данных
Новое поле в `user_settings`:
```
food_forward_recipients JSONB DEFAULT '[]'   -- list[int] chat_id получателей
```
- Миграция: `database/migrations/add_food_forward_recipients.sql`
- Модель: `UserSettings.food_forward_recipients` (`database/models.py`)
- Пусто по умолчанию → фича выключена для всех, кроме явно настроенных.

## Поток
```
Пользователь шлёт фото еды → распознавание КБЖУ → «✅ Сохранить»
   → handle_meal_confirmation (callback) → save_meal_to_db OK
   → forward_meal_to_recipients(bot, sender_id, meal_name, totals, photo_path)
        → для каждого chat_id из food_forward_recipients:
             send_photo(фото, caption="🍽 Андрей: <блюдо> · ккал · Б/Ж/У")
```

## Точка интеграции
`telegram-bot/handlers/photo.py` → `handle_meal_confirmation`, **после успешного `save_meal_to_db`** (тот же колбэк обслуживает и меню, и обычные блюда — единая точка). Хук обёрнут в `try/except` — вспомогательный поток никогда не ломает сохранение еды.

Фото берётся из состояния: `photo_path` (ветка меню) или первый из `photo_paths` (ветка описания). Если фото нет (приём залогирован текстом) — отправляется только текст с КБЖУ.

## Модуль
`core/family/forward.py`:
- `get_recipients(sender_id) -> list[int]` — читает из UserSettings.
- `forward_meal_to_recipients(bot, sender_id, meal_name, totals, photo_path, sender_name) -> int` — рассылает, возвращает число доставленных. Не бросает исключений наружу.

## Ограничение Telegram (важно)
Бот **не может писать первым** — каждый получатель должен **сам нажать `/start`** у @Botkin_md_bot. Если получатель не начинал диалог → `TelegramForbiddenError` ловится и логируется, рассылка остальным продолжается.

## Управление получателями (команды бота)
Реализовано в `telegram-bot/handlers/commands.py`:
- `/forward` — показать список получателей
- `/forward_add` — добавить (ответом на пересланное сообщение от человека или по chat_id)
- `/forward_remove <chat_id>` — удалить

⚠️ Боты Telegram не умеют резолвить `@username` → добавление по **chat_id** (или через пересланное сообщение, откуда бот берёт `forward_from.id`). Получатель должен заранее нажать `/start`.

## Тест-план
- [ ] Миграция применяется идемпотентно (`IF NOT EXISTS`).
- [ ] Получатель с `/start` получает фото + корректный caption.
- [ ] Получатель без `/start` → `TelegramForbiddenError` залогирован, остальные получили, сохранение еды не сломалось.
- [ ] Приём без фото (текст) → получатель получает только КБЖУ-текст.
- [ ] Пустой `food_forward_recipients` → ничего не отправляется (no-op).
- [ ] Медданные (глюкоза/давление/диагнозы) НЕ утекают — пересылается только meal_name + totals.

## Возможные доработки (вне этого PR)
- Дневной дайджест вместо каждого приёма (если получатель не хочет поток сообщений) — актуально для врача.
- Согласие получателя (opt-in кнопка при первом форварде).
- Резолв `@username` через хранение mapping username→chat_id при /start.
