# Редактор дневника питания в Telegram Mini App

**Дата:** 2026-04-17
**Автор:** Александр Лысковский (brainstorm с Claude)
**Статус:** design approved, pending implementation plan

## Цель

Дать пользователю (Александру, `user_id=895655`) возможность **просматривать и редактировать** всё, что съедено в любой день, прямо из Telegram. Сейчас приёмы пищи вносятся только через NutriLogBot (фото/текст/голос), но редактирование и ретроспективный просмотр невозможны — надо открывать БД.

## Проблема

- После ввода в бота нельзя исправить вес продукта («съел не 100, а 150 г»).
- Нельзя увидеть структуру дня одним экраном.
- Нельзя удалить ошибочно распознанный продукт.
- Нет обратной связи по прогрессу до целей в течение дня.

## Решение: Day Editor Screen

Вторая страница внутри существующего мини-аппа `telegram-bot/webapp/index.html` (vanilla HTML/CSS/JS, без билд-шага). Дневник становится **главным** экраном мини-аппа, настройки переносятся в подраздел (иконка ⚙️ в углу).

### Key UX decisions (с обоснованием рыночных референсов)

Ресерч (отдельный агент, см. transcript) показал, что три топ-трекера — **MyFitnessPal (~200M), Yazio (~60M), Cronometer (~10M)** — сошлись на одинаковом паттерне. Наш дизайн повторяет этот паттерн, с одним отклонением в сторону Cronometer (бары БЖУ+клетчатка вместо «осталось ккал»), что уместно для health-трекера, а не diet-трекера.

| # | Решение | Источник |
|---|---|---|
| 1 | Day-switcher сверху `‹ Сегодня ›` + календарь | единогласно |
| 2 | 4 фиксированных слота (Завтрак / Обед / Ужин / Перекус), пустые видны | единогласно |
| 3 | Слоты свёрнуты по дефолту, раскрытие тапом | Yazio, Cronometer |
| 4 | `+ добавить` внутри каждого слота, без глобального FAB | большинство |
| 5 | Свайп влево → Delete (iOS-native) | MFP, Yazio |
| 6 | Футер: прогресс-бары ккал / Б / Ж / У / клетчатка до целей | Cronometer |
| 7 | Bottom sheet для добавления: favorites + ручной ввод | MFP, Yazio |

## Архитектура

### Фронт

Расширяем существующий `telegram-bot/webapp/index.html`:
- Секция «Дневник» = главный экран (показывается при открытии).
- Секция «Настройки» = текущее содержимое, переезжает в подраздел. Доступ — иконка ⚙️ в шапке.
- Vanilla JS, никакого билда, никакого React.
- CSS-переменные Telegram (`--tg-theme-*`) уже используются — поддержка тёмной темы «бесплатно».

### Бэк

Новые HTTP-эндпоинты в том же aiogram-процессе, файл `telegram-bot/webhook/nutrition_api.py`. Работают рядом с существующим `apple_health.py`.

### Авторизация

Telegram WebApp `initData` → проверка HMAC-SHA256 по `BOT_TOKEN` (стандартная TMA-схема) → извлечение `user_id`. Все запросы скоупированы по `user_id`. Запросы без валидного `initData` возвращают 401.

### Таймзона

Europe/Moscow на бэке и на фронте. Дата передаётся строкой `YYYY-MM-DD`. Сервер вычисляет текущую дату через `datetime.now(ZoneInfo("Europe/Moscow")).date()`.

### Элементы

1. **Header:** название «🍽 Дневник» + кнопка ⚙️ (переключает на секцию настроек).
2. **Day-switcher:** стрелки `‹` / `›` (±1 день), заголовок («Сегодня» / «Вчера» / «Завтра» / «17 апреля, пт»), кнопка «📅 Выбрать дату» открывает native `<input type="date">`. Свайп-жест left/right по всему экрану = переключение дня (опционально, если хватит времени).
3. **Слоты:** карточка с закруглением 14px, фон `#f7f7fa` (светлая) / tg-theme-secondary (тёмная). Пустые слоты с классом `dim`: серый текст + `+` справа. Заполненные: эмодзи + название + время + сумма ккал + индикатор `›` (свёрнут) / `⌄` (раскрыт).
4. **Продукты внутри раскрытого слота:** строка с двумя уровнями — верхний (имя + граммовка справа), нижний (БЖУ+клетчатка+ккал мелким кеглем серым цветом). Разделитель — пунктирная линия.
5. **Свайп-to-delete:** CSS translate на touchmove, красная «Удалить» подложка. Onboarding-подсказка «← свайп для удаления / тап для редактирования» показывается **один раз** при первом раскрытии слота (флаг `localStorage.swipeHintShown`).
6. **Тап по продукту:** открывает bottom-sheet-редактор с одним полем «Вес, г». Сохранение → PATCH, КБЖУ пересчитываются пропорционально.
7. **Футер:** sticky, 5 прогресс-баров. Цвета: ккал #2a7ae2, Б #e0577c, Ж #f5b02b, У #4cb563, клетчатка #8e79d6. Если цель не задана в настройках — показываем только цифру без бара.

### Add bottom sheet

Открывается по «+ добавить в {slot}» или по кнопке `+` в заголовке пустого слота.

```
┌──────────────────────────────────────┐
│          Добавить в Обед          ✕  │
├──────────────────────────────────────┤
│  Часто используемое:                 │
│  ┌────────────┬────────────┐         │
│  │ Кофе       │ Протеин    │ …      │   ← горизонтальный скролл чипов
│  │ 200 мл     │ 30 г       │         │     (последние 15 из user_products)
│  └────────────┴────────────┘         │
│                                      │
│  Или ввести вручную:                 │
│  ┌────────────────────────────────┐  │
│  │ Название                       │  │
│  └────────────────────────────────┘  │
│  ┌──────┐                            │
│  │ Вес г│                            │
│  └──────┘                            │
│                                      │
│  ┌────────────────────────────────┐  │
│  │        Добавить                │  │   ← MainButton Telegram API
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

- Chips из `user_products` (уже существует таблица) — тап добавляет с `default_weight` без вызова LLM (КБЖУ известны per-100g в записи).
- Ручная форма — тап «Добавить» → спиннер → POST `/api/meal/item` с `source=manual`, бэк зовёт существующий LLM-пайплайн из `core/nutrition/`.
- Таймаут 10 сек на LLM. Если упал — продукт всё равно сохраняется с `p=f=c=fib=0` и красным индикатором «❗ КБЖУ не распозналось, тапни чтобы задать вручную».

## API

Все эндпоинты требуют валидный `initData` (HMAC-SHA256 с `BOT_TOKEN`) в заголовке `X-Telegram-Init-Data`. `user_id` извлекается из проверенного initData.

### GET /api/day

**Query:** `date=YYYY-MM-DD`

**Response 200:**
```json
{
  "date": "2026-04-17",
  "meals": [
    {
      "id": 12345,
      "meal_name": "Завтрак",
      "meal_time": "09:10",
      "slot": "breakfast",
      "items": [
        {"idx": 0, "name": "Овсянка на молоке", "weight": 80, "p": 8, "f": 4, "c": 46, "fib": 6, "kcal": 260}
      ],
      "totals": {"kcal": 520, "p": 22, "f": 14, "c": 68, "fib": 8}
    }
  ],
  "totals_day": {"kcal": 1200, "p": 82, "f": 32, "c": 88, "fib": 9},
  "goals": {"kcal": 1920, "p": 150, "f": 66, "c": 220, "fib": 30}
}
```

`slot` вычисляется сервером из `meal_name` (fuzzy-match: «завтрак», «breakfast», «🌅 Завтрак») или из `meal_time` если имя не матчится:
- 06:00–10:59 → breakfast
- 11:00–14:59 → lunch
- 15:00–17:59 → snack
- 18:00–05:59 → dinner

### POST /api/meal/item

**Body:**
```json
{
  "date": "2026-04-17",
  "slot": "lunch",
  "name": "Курица грудка запечённая",
  "weight": 180,
  "source": "manual"
}
```

`source` ∈ `"manual"` (LLM из name) | `"user_product"` (в этом случае передаётся `user_product_id`, КБЖУ берутся из таблицы с масштабированием на вес).

**Логика:**
1. Ищем существующий `nutrition_log` за `date` + `user_id` где `meal_name` матчится слот. Если найден — добавляем в его `items[]`, пересчитываем `totals`.
2. Иначе создаём новую запись `nutrition_log`:
   - `meal_name` = «Завтрак» / «Обед» / «Ужин» / «Перекус»
   - `meal_time` = центр слота: 09:00 / 13:00 / 19:00 / 16:00
3. Если `source=manual` — зовём `core/nutrition/...` с name → получаем КБЖУ на weight.
4. Также upsert в `user_products` (обновляем `last_used_at`, создаём запись если новый продукт).

**Response 201:**
```json
{"meal_id": 12346, "item": {"idx": 3, "name": "...", "weight": 180, "p": 38, "f": 6, "c": 0, "fib": 0, "kcal": 218}}
```

### PATCH /api/meal/item

**Body:** `{"meal_id": 12345, "idx": 2, "weight": 200}`

Меняет **только** вес. КБЖУ пересчитываются пропорционально: `new_p = old_p * new_weight / old_weight`. `totals` приёма обновляются.

**Response 200:** обновлённый item + новые `totals`.

### PATCH /api/meal

**Body:** `{"meal_id": 12345, "meal_name": "Поздний обед", "meal_time": "15:30"}`

Оба поля опциональны. Смена `meal_time` может изменить принадлежность к слоту при следующем GET.

### DELETE /api/meal/item

**Query:** `meal_id=12345&idx=2`

Удаляет продукт из `items[]`. Если это был последний item — удаляет приём целиком (`nutrition_log` row). Пересчитывает `totals` или возвращает 204 без тела.

Фронт хранит удалённый item в памяти 4 секунды и показывает snackbar «Удалено · Отменить». Отмена = повторный POST /api/meal/item.

### DELETE /api/meal

**Query:** `meal_id=12345`

Удаляет весь приём.

### GET /api/favorites

**Query:** `limit=15` (дефолт 15, максимум 50)

**Response 200:**
```json
[
  {
    "user_product_id": 42,
    "name": "Кофе с молоком",
    "default_weight": 200,
    "per_100": {"p": 3, "f": 3, "c": 4, "fib": 0, "kcal": 55},
    "last_used": "2026-04-17T08:45:00+03:00"
  }
]
```

Сорт по `last_used_at DESC`. Используется для chips в bottom sheet.

## Edge cases

| Случай | Поведение |
|---|---|
| Пустой день | 4 dim-слота с `+`, футер с нулями и целями. |
| Будущая дата | `›` разрешает до **+7 дней** (план на неделю). После +7 — кнопка disabled. |
| Прошлое без ограничений | `‹` без ограничений — можно уйти в любую дату, в которой есть записи. |
| 2+ приёма в один слот | Показываем обе карточки подряд под заголовком слота («Обед 12:30», «Обед 14:00»). Каждая редактируется независимо. |
| LLM timeout / fail | Продукт сохраняется с `p=f=c=fib=0`, красный индикатор `❗`. Тап → ручной ввод КБЖУ в bottom sheet. |
| initData invalid/missing | 401 Unauthorized, фронт показывает «Откройте через бота». |
| Цель не задана | Цифра без бара, без процента. |
| Swipe accidental | Snackbar «Удалено · Отменить» 4 сек. |
| Offline / нет сети | Toast «Нет связи, попробуй ещё раз». Без offline-first режима. |
| Сайд-эффект: новый LLM-продукт | upsert в `user_products`, попадёт в favorites со следующего открытия. |

## Непокрытое (out of scope для первой итерации)

- **Фото-сканирование из мини-аппа** — остаётся в боте. Возможно добавим Phase 2.
- **Поиск по USDA / OpenFoodFacts каталогу** — отдельная большая инфраструктура.
- **Штрих-код** — не нужен (Александр готовит сам, упаковок мало).
- **Голосовой ввод в мини-аппа** — остаётся в боте.
- **Редактирование КБЖУ вручную** (помимо fallback при LLM-fail) — пересчёт через вес покрывает 90%.
- **Множественное удаление** — не нужно на масштабе пользователя.
- **Копирование приёма на другой день** — возможно Phase 2.
- **Статистика за неделю/месяц** — отдельный экран, отдельный спек.

## Технический стек

| Слой | Технология |
|---|---|
| Фронт | Vanilla HTML5 + CSS + JS, Telegram WebApp SDK (`telegram-web-app.js`) |
| Бэк | Python 3 + aiogram (существующий процесс), новый модуль `telegram-bot/webhook/nutrition_api.py` |
| БД | PostgreSQL (существующая), таблицы `nutrition_log`, `user_products`, `users` |
| LLM | Существующий пайплайн `core/nutrition/` (OpenAI / Gemini, уже настроен) |
| Авторизация | Telegram `initData` HMAC-SHA256 |
| Деплой | Существующий `docker-compose.prod.yml`, один контейнер |
