## 📁 **Структура данных**

### **Основные источники данных для Александра:**
```
📱 Apple Health → data/apple-health/export.xml
   ├── Zepp Life (умные весы): вес, жир%, тощая масса
   ├── Connect/Garmin: тренировки, активность, сон
   ├── Sleep Cycle: детальный анализ сна  
   └── HealthVault Tracker: питание через бота

🏋️ Тренировки → Apple Health Workouts + активности
   ├── Кроссфит: записывается как HIIT на Garmin
   ├── Контора: Пн/Ср/Пт 19:00-20:00
   └── МФР/Восстановление: Суббота ~12:00-13:00

🍽️ Питание → data/nutrition/nutrition_log.json
   ├── Telegram бот обработка
   ├── КБЖУ автоматический расчет
   └── Детальная разбивка по приемам пищи

💊 Витамины → data/logs/supplements/log.json
   └── Протокол HealthVault v1
```

## 🔍 **Диагностика проблем**

### **Если анализ показывает странные результаты:**

#### **"Нет тренировок"**
```bash
# Проверить Apple Health Workout данные
python3 -c "
import xml.etree.ElementTree as ET
context = ET.iterparse('data/apple-health/export.xml', events=('end',))
workouts = [elem for event, elem in context if elem.tag == 'Workout']
print(f'Найдено Workout записей: {len(workouts)}')
"
```

#### **"Нет данных состава тела"**
```bash
# Проверить записи от Zepp Life
grep -c 'sourceName="Zepp Life"' data/apple-health/export.xml
# Должно быть > 500 записей
```

#### **"Странные данные питания"**  
```bash
# Проверить неполные дни
python3 -c "
import json
from datetime import datetime
with open('data/nutrition/nutrition_log.json') as f:
    data = json.load(f)
for entry in data['entries'][-7:]:  # Последние 7 дней
    if entry['totals']['calories'] < 1200:
        print(f\"Подозрительный день: {entry['date']} - {entry['totals']['calories']} ккал\")
"
```

## 🛠️ **Инструменты диагностики**

### **1. Валидатор данных**
```bash
python3 scripts/validate_health_data.py
```
- Проверяет полноту всех источников
- Выявляет неполные дни
- Дает рекомендации по исправлению

### **2. Исправленный анализатор**  
```bash
python3 scripts/fixed_apple_health_analyzer.py
```
- Детальная отладка парсера
- Поиск пропущенных данных
- Сопоставление с ожидаемыми значениями

### **3. Быстрая проверка в терминале**
```bash
# Подсчет записей по источникам
grep -o 'sourceName="[^"]*"' data/apple-health/export.xml | sort | uniq -c

# Подсчет типов данных
grep -o 'type="[^"]*"' data/apple-health/export.xml | sort | uniq -c | head -20
```

## 📈 **Процедура корректного анализа**

### **Шаг 1: Подготовка**
1. Обновить экспорт Apple Health (если старше 3 дней)
2. Проверить синхронизацию всех приложений
3. Запустить валидатор данных

### **Шаг 2: Парсинг** 
1. Запустить основной парсер
2. Проверить статистику по источникам
3. При проблемах - запустить исправленный анализатор

### **Шаг 3: Анализ**
1. Исключить неполные дни
2. Указать источник каждого вывода  
3. Отметить ограничения данных
4. Сделать выводы только на основе проверенных данных

### **Шаг 4: Отчет**
1. Начать с валидации данных
2. Указать период и исключения
3. Привести детальную статистику
4. Дать рекомендации с учетом ограничений
